# Техническое задание GoCoursesLab — ревизия v2

**Стек (ориентир):** Expo SDK 54, React Native, expo-router (`app/`), SQLite (`expo-sqlite`), i18next (ru/en), тема light/dark/system + AsyncStorage, нативный сплэш `expo-splash-screen` с удержанием до готовности приложения.

---

## 1. Общее

### 1.1. Цель релиза

Поставить учебное мобильное приложение «лабораторная платформа курсов»: навигация по трём основным разделам, локализация, тема с сохранением, предсказуемый холодный старт со сплэшем и **проверяемый** локальный CRUD над данными в SQLite по единой согласованной модели (см. разд. 3).

### 1.2. Целевые платформы и окружение приёмки

| Уровень | Окружение | Назначение |
|--------|-----------|------------|
| **Основное** | Android-эмулятор (рекомендуется API 33+) **или** физическое устройство Android с **Expo Go** / dev client | Полная проверка п. 1–5, включая SQLite |
| **Дополнительное** | **Web** (режим `expo start --web` или статическая сборка из `app.json`) | Проверка UI/навигации/i18n/темы; поведение БД — по реализации проекта (при наличии веб-адаптера хранилища) |

Все критерии приёмки ниже по умолчанию выполняются в **основном** окружении, если в колонке «Примечание по окружению» не указано иное.

### 1.3. Вне scope (не являются обязательными критериями базового ТЗ)

Ниже перечислен фактический объём «надстройки» над ядром ТЗ; он может присутствовать в коде, но **не заменяет** требования п. 1–5 и не требует отдельной формальной приёмки в v2, если не оговорено заказчиком:

- Каталог учебного контента из **статического мока** (`MOCK_COURSES`): превью, описания, вложенные тесты.
- Прогресс прохождения тестов, голоса «лайк/дизлайк», «последний курс» — **локальная персистентность через AsyncStorage** (ключи вида `mock:…`), не путать с проверкой п. 5.
- Роль «создатель курсов» и отдельные экраны редактирования сущностей вне выбранной для п. 5 модели — **вне scope**, если явно не включены в договорённость.

## 2. Требования (переработанные п. 1–5)

### 2.1. П. 1 — Пользовательский интерфейс: «страницы» и сценарии

**Определение «страницы»:** полноэкранный маршрут верхнего уровня. В проекте **минимум три таких страницы** реализованы как **корневые вкладки** tab-навигатора: «Главная», «Каталог», «Настройки».

**Дочерние маршруты** (stack, модальные экраны — `course/[id]`, редакторы, вспомогательные списки) **не считаются** отдельными «страницами» для выполнения минимума «3 страницы», но **подчиняются** требованиям локализации и темы там, где отображается пользовательский текст.

**Пользовательские сценарии (1–2 обязательных для демонстрации):**

1. **Навигация по вкладкам:** с «Главной» перейти в «Каталог», открыть курс, вернуться; открыть «Настройки» и вернуться на «Главную».
2. **Работа с локальными данными по п. 5:** открыть сценарий CRUD выбранной сущности (см. разд. 3) и выполнить создание → просмотр в списке → редактирование → удаление (шаги зафиксированы в таблице приёмки).

### 2.2. П. 2 — Экран загрузки и сплэш (два уровня)

Требование разделено на **нативный** сплэш и **экран инициализации** до состояния `ready`.

**(а) Нативный сплэш** задаётся конфигурацией Expo Plugin `expo-splash-screen` в `app.json` (изображение, фон светлой/тёмной темы, `userInterfaceStyle: automatic`). Он показывается **с момента запуска нативной оболочки** до передачи управления JS и до явного скрытия сплэша из кода.

**(б) Логика приложения:** вызывается `SplashScreen.preventAutoHideAsync()` при старте; пока приложение **не готово** (`ready === false`), пользователь видит **экран инициализации** (boot): локализованный текст и визуальный индикатор загрузки (например, анимация). После успешной инициализации (язык, тема из хранилища, шрифты, инициализация SQLite и i18n согласно `AppProvider`) устанавливается `ready === true`, вызывается `SplashScreen.hideAsync()`, отображается основная навигация.

**Однозначный критерий холодного старта:** при полном перезапуске приложения сначала виден **нативный** сплэш, затем при необходимости **экран boot**, затем — основной интерфейс; не допускается «мигание» пустого root без сплэша до первого кадра boot (ожидаемое поведение при использовании `preventAutoHideAsync` до `hideAsync`).

### 2.3. П. 3 — Локализация

**Языки:** русский и английский; переключение доступно пользователю (экран «Настройки»), выбор сохраняется между сеансами (AsyncStorage).

**Охват:** все **пользовательские строки**, отображаемые на **экранах и модалках**, должны проходить через i18next (ключи в ресурсах `ru` / `en`). Допускается `defaultValue` у ключа только как временная мера; **критерий приёмки** — отсутствие **хардкода** видимого текста на **основных** экранах вкладок, boot, настройках, экране курса и **экранах, относящихся к п. 5** (список/редактор/детали выбранной сущности).

*Известный риск (на момент ревизии ТЗ):* отдельные маршруты могут содержать строки вне i18n — это **недопустимо** для соответствия настоящему ТЗ v2 и подлежит выправлению при приёмке.

**Проверка:** смена языка в настройках немедленно меняет язык интерфейса на всех открытых экранах после перерисовки; после перезапуска приложения язык сохраняется.

### 2.4. П. 4 — Тема интерфейса

**Режимы:** в ТЗ **явно разрешены три варианта** предпочтения темы: **светлая**, **тёмная**, **системная** (следование настройкам ОС). Поведение **резолвинга**: при «системной» теме итоговая схема (`light` / `dark`) определяется системным `useColorScheme` на момент рендера.

**Персистентность:** выбранное предпочтение (`light` | `dark` | `system`) сохраняется в AsyncStorage и восстанавливается после полного перезапуска приложения.

**Не требуется** сводить продукт к «строго двум режимам» без системного — три режима являются нормативными для v2.

### 2.5. П. 5 — Локальная база данных (SQLite)

Выбрана **одна согласованная линия** проверки (согласно замечанию ревьюера — без смешения «мок + SQLite + AsyncStorage» в одном пользовательском сценарии заметок):

**Нормативная сущность для п. 5 — «Заметка по курсу»** в таблице SQLite `notes` (или эквивалентное имя в коде при сохранении совместимости схемы).

**Поля:**

| Поле | Тип по смыслу | Пользователь в UI |
|------|---------------|-------------------|
| `id` | целое, PK, автоинкремент | не редактирует; видит как идентификатор/не обязательно отображать |
| `courseId` | целое, связь с курсом | выбор курса контекстом экрана (id из мок-каталога); в форме может быть скрыт, но логический контекст обязателен |
| `title` | строка | **создаёт и редактирует** |
| `body` | строка | **создаёт и редактирует** |
| `status` | перечисление (`planned` / `in_progress` / `done`) | **выбирает** |
| `updatedAt` | целое (Unix ms), служебное | **обязательно отображается** при просмотре списка или карточки заметки (read-only, формат даты/время — по UX проекта) |

**Требование «≥ 3 полей»:** выполняется за счёт **трёх редактируемых пользователем** полей (`title`, `body`, `status`). Поля `id`, `courseId`, `updatedAt` — служебные; **`updatedAt` не может быть единственным «третьим полем», скрытым от пользователя** — дата/время изменения должна быть **видна** при приёмке.

**Источник каталога курсов для контента:** перечень курсов с уроками/тестами — **мок** (`MOCK_COURSES`). Поле `courseId` в заметке ссылается на **id** из этого каталога. Это **не освобождает** от полноценного SQLite CRUD по таблице заметок в UI.

**Запрещено для приёмки п. 5:** единственный канал «заметки» только через AsyncStorage или только через демонстрацию API без доступного из UI маршрута создания/редактирования/удаления заметок в SQLite.

**Сквозная связь UI ↔ SQLite:** создание, чтение списка, обновление, удаление выполняются через слой `expo-sqlite`; после операций список/экран деталей отражает актуальное состояние БД.

## 3. Модель данных для п. 5

### 3.1. Сущность «Заметка по курсу» (`CourseNote`)

- **Таблица:** `notes`
- **Минимальный набор колонок:** `id`, `courseId`, `title`, `body`, `status`, `updatedAt`
- **Индексы:** допустим индекс по `courseId` для списков

### 3.2. Отображение в списке

В списке заметок (по выбранному курсу или общий список — по реализации) пользователь видит минимум: **заголовок**, **статус** (или его локализованную метку), **время изменения** (`updatedAt`). Тело заметки — в детальном просмотре и/или в редакторе.

### 3.3. Источник каталога курсов

- **Каталог карточек курсов (превью, тесты):** статический мок; **не** является строкой таблицы `courses` для целей п. 5.
- **Таблица `courses` в SQLite** (если присутствует в коде): относится к **расширению / роли создателя** и для базовой приёмки **не заменяет** проверку п. 5 по `notes`, если иное не согласовано отдельно.

### 3.4. Согласование с возможным дублированием в коде

Если в интерфейсе сохраняется **вторая** модель «одна текстовая заметка на курс» в AsyncStorage, для соответствия ТЗ v2 требуется **одна пользовательская правда для п. 5** — SQLite `notes`; дублирующий UX должен быть удалён, спрятан за флагом разработки или явно вынесен **вне** сценария приёмки (договорённость в «Открытых вопросах»).

## 4. Критерии приёмки

| № | Требование (сводно) | Шаги проверки | Ожидаемый результат | Примечание по окружению |
|---|---------------------|---------------|---------------------|-------------------------|
| AC-1 | ≥ 3 «страницы»-вкладки | Открыть приложение; переключить три нижние вкладки | Видны три разных корневых экрана: Главная, Каталог, Настройки | Android / Expo Go |
| AC-2 | Сценарий навигации | С «Главной» → «Каталог» → открыть курс → системная «Назад» → «Настройки» | Навигация без краша; контент соответствует маршрутам | Android / Expo Go |
| AC-3 | Нативный сплэш | Cold start (убить процесс, запустить снова) | Сначала нативный сплэш из конфига Expo; затем при необходимости boot UI | Android / Expo Go |
| AC-4 | Boot до `ready` | При медленном устройстве или логировании времени инициализации | Пока `ready === false`, не показывается основной Stack/Tabs; после готовности сплэш скрыт, UI доступен | При необходимости отладочный лог |
| AC-5 | Локализация ru/en | Настройки → сменить язык; пройти вкладки и экран п. 5 | Весь видимый пользовательский текст соответствует выбранному языку; нет непереведённых фрагментов на контрольных экранах | ru и en проверить оба |
| AC-6 | Сохранение языка | Выбрать `en`, force-stop, запуск | Интерфейс на английском | Android |
| AC-7 | Тема: три режима | В настройках выбрать **Светлая** / **Тёмная** / **Системная** | UI перекрашивается согласно режиму; системная — следует ОС | Android |
| AC-8 | Сохранение темы | Выбрать **Тёмная**, force-stop, запуск | Тема остаётся тёмной (предпочтение `dark` восстановлено) | Не проверять одновременно с системной |
| AC-9 | Системная тема | Установить «Системная»; сменить тему ОС (если доступно) | Приложение следует системной схеме | Зависит от устройства |
| AC-10 | П.5: CREATE | Из UI открыть создание заметки; заполнить title, body, status; сохранить | Запись появилась в SQLite; отображается в списке с `updatedAt` | Android (доп. Web по адаптеру) |
| AC-11 | П.5: READ | Открыть список заметок по курсу (или общий список) | Видны все сохранённые поля согласно разд. 3.2 | |
| AC-12 | П.5: UPDATE | Отредактировать заметку, сохранить | Изменения в списке/детали; `updatedAt` обновлён | |
| AC-13 | П.5: DELETE | Удалить заметку подтверждением или кнопкой удаления | Запись исчезла из UI и из выборки БД | |
| AC-14 | П.5: не только служебные поля | Открыть карточку заметки | Видны **заголовок**, **текст**, **статус**, **дата изменения** | Запрещён сценарий «только timestamp в БД без отображения» |

## 5. Открытые вопросы (для заказчика / преподавателя)

1. Подтвердить, что **единственный** проверяемый CRUD по п. 5 — таблица **`notes`** в SQLite, а **не** отдельная демонстрация только таблицы `courses` без заметок.
2. Нужна ли в рамках учебной сдачи **формальная** приёмка **Web** (п. 5), если используется адаптер не-SQLite, или достаточно зафиксировать Web как опциональный просмотр без БД?
3. Следует ли **удалить** или **заблокировать** UX «одно текстовое поле заметки на курс» через AsyncStorage на экране курса, чтобы исключить дублирование с SQLite, или допускается оставить как устаревший путь вне приёмки?
4. Требуется ли включить в scope роль «создатель» и редактирование **`courses`** в SQLite с выводом в каталог, или каталог **навсегда** остаётся моком для учебного стенда?
5. Достаточно ли для отображения `updatedAt` **относительного** формата («5 мин назад») или нужен строго **абсолютный** локализованный таймстамп?
